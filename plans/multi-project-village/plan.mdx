# Codeville — Five-Project Village

Codeville becomes a small working town rather than one isolated workshop. Five project lots remain visible at once, each with its own builder, task, phase, progression, approval state, and completion debrief. A developer can start work in one project, move to another while it runs, and understand the whole village without reading five terminal logs.

The visible animation stays truthful: every builder transition comes from that project's real Codex thread. The game layer keeps one Pixi application alive and schedules event-driven movement so rapid protocol updates feel continuous instead of destroying and rebuilding the scene.

<Callout id="root-cause" tone="warning">

**Confirmed animation root cause:** `VillageCanvas` currently places `phase`, `level`, and `projectName` in the Pixi initialization effect's dependency array. Every Codex event destroys the Pixi application, replaces the canvas, resets the builder to its spawn point, and discards interpolation/effect state. A live reproduction proved the original canvas survived neither the first planning event nor the next action. The new scene must initialize once and accept state updates through a stable controller.

</Callout>

## Product model

The village has exactly five lot slots for the hackathon release. A slot can be empty, idle, active, waiting for approval, completed, failed, or interrupted. All five lots remain visible; selection changes the right-hand Foreman's Desk, not the map. This gives the product a stable spatial memory: Acorn Tasks is always on the same lot, even while another project is selected.

The one-click demo provisions five safe copies of the existing fixture with distinct names. Each copy is a real repository and can start a real Codex thread. User-owned repositories can fill any lot and start individually. Checkboxes on real-project cards feed a visually distinct **Start selected builders** control; it opens a prominent exact name/path/task preflight, and cancellation launches nothing. The disposable **Start all builders** action remains visually separate and now uses the same confirmation gate.

## Completion debrief

The agent's last visible action is a small speech bubble anchored above its builder. It has two short fields:

- **Landed:** a plain-language outcome, for example “Health summary is working and all checks pass.”
- **Follow-up:** either one recommended next step or “No follow-up recommended.”

The same content appears in the selected project's panel and persists locally. The Codex thread receives a developer instruction to append a discriminated machine-readable `CODEVILLE_RESULT` object. The main process validates either a completed debrief or a bounded waiting question. Missing or unsafe markers become `needs_review`, never verified success. Raw agent prose never reaches the renderer and is discarded immediately after parsing.

<Callout id="debrief-decision" tone="decision">

**Debrief contract:** treat the final message as untrusted input. The renderer receives only `{ landed, followUp, followUpRecommended }` after local parsing and sanitization. Maximum 96 characters per text field; reject path separators, URLs, email addresses, backticks, shell sigils, common secret prefixes, and code-shaped identifiers. A rejected debrief becomes “Improvement completed.” plus a test-derived follow-up default.

</Callout>

### Session ownership

`electron/main.ts` replaces the singleton `activeSession` with two maps: `sessionsByThreadId` and `threadIdByProjectId`. Notification and approval params already carry `threadId`, so routing is deterministic. A project cannot start a second turn while its own turn is active, but other projects remain independent. Interrupt takes `projectId`; approval responses preserve both `requestId` and `projectId`.

The single `AppServerClient` remains the transport owner. JSON-RPC request correlation already supports concurrent requests, so the new work is project ownership and cleanup rather than five subprocesses. Each runtime is registered immediately after `thread/start` and before `turn/start`; a failed start removes only that runtime. Completed `agentMessage` items are accumulated privately by `threadId` so debrief extraction does not depend on the terminal turn payload carrying a full item view.

<Callout id="concurrency-proof" tone="success">

**Installed-runtime proof, July 17:** one Codex 0.144.4 app-server completed two simultaneous `gpt-5.6-sol` threads over the existing JSONL stdio transport. The probe received two distinct thread ids, the correct independent final messages, two terminal completions, and 28 notifications routed across both ids. This confirms that Codeville should multiplex threads through one initialized connection. The [official app-server lifecycle](https://learn.chatgpt.com/docs/app-server.md) also defines thread-scoped turns, streamed item notifications, and terminal `turn/completed` events.

</Callout>

<Callout id="debrief-runtime-proof" tone="success">

**Result protocol proof:** Codeville retains only the latest private agent-message text per active thread, parses the required `CODEVILLE_RESULT` at terminal completion, emits only its sanitized projection, and immediately discards the raw accumulator. Deterministic protocol coverage proves validated completion, terminal waiting, malformed-result review, and same-thread continuation.

</Callout>

## Smooth gameplay model

`VillageCanvas` initializes `Application`, `VillageScene`, five `ProjectLot` containers, and five `BuilderActor` controllers once. A second effect calls `scene.update(snapshot)` without touching the canvas or GPU resources.

Each `BuilderActor` holds independent visual state:

```ts
interface BuilderVisualState {
  displayedPhase: SessionPhase;
  queuedPhases: SessionPhase[];
  enteredAtMs: number;
  position: Point;
  velocity: Point;
  target: Point;
}
```

The scheduler follows four rules:

1. Deduplicate repeated phase events from reasoning and item deltas.
2. Keep planning/reading/editing/testing visible for at least 650 ms unless approval or failure preempts them.
3. When more than three phases queue, coalesce adjacent planning/reading states but retain editing, testing, approval, and terminal outcomes.
4. Move with a delta-time critically damped spring and animate gait from actual velocity. Never set an actor back to its spawn point on phase change.

Completion waits behind already-observed work phases, then moves the builder to its doorstep, upgrades the lot, and eases the bubble in. Visual choreography may trail the logical sidebar state by at most two seconds; terminal events skip stale low-value planning/reading entries when needed to stay inside that bound. The bubble remains until that project starts another task or the user dismisses it; it does not expire while the user is reading another project.

Project task text is never persisted; per-project drafts live only in renderer memory and disappear at relaunch. Paths and names remain local in the mode-`0600` atomic store. Project ids are opaque local UUIDs and never derived from or reversible to a filesystem path. Project records retain repository metadata even when displaced from a lot, so reopening the same path restores identity, progression, and safe debrief. The migration creates one UUID for each version 1 path key, assigns migrated projects to available lots, preserves every prior progression record, and leaves remaining lot slots empty.

## Ordered implementation

1. **Lock contracts first.** Add the version 2 village schema, v1 migration, `ProjectVillageEvent`, `CompletionDebrief`, and pure tests. No renderer or protocol code moves until privacy and migration invariants pass.
2. **Route real concurrency.** Replace singleton session ownership in `electron/main.ts`, include `projectId` in every renderer envelope, make interrupt/approval project-scoped, and prove interleaved notifications cannot cross projects.
3. **Create the five-project demo and real-project launch surface.** Provision five fixture copies, persist real/demo assignments, add individual start, selectable real-project subsets, and exact confirmation for every aggregate launch.
4. **Build discriminated results and interaction.** Parse completed/waiting results, route native input, sanitize and persist only safe projections, retain saved thread identity, and test malicious/path/code/secret/oversized payloads plus missing markers.
5. **Replace the game runtime.** Split scene, lot, and actor ownership. Mount Pixi once, update via controller, add phase queues and spring movement, and render anchored debrief bubbles.
6. **Recompose the desktop shell.** Keep the present brand/topbar, expand the stage to the five-lot ward, add the project rail and selected-project desk, and preserve readable state at the 1080 px minimum width.
7. **Prove gameplay.** Add a canvas identity regression, deterministic scheduler tests, fake-clock actor motion assertions, five-stream renderer tests, and screenshot/animation sampling for overlap and visual continuity.
8. **Prove the real product.** Run two concurrent real Codex repairs in the default acceptance suite and run the same harness with `CODEVILLE_E2E_PROJECT_COUNT=5` as the release gate. Verify five isolated repositories, five routed debriefs, relaunch, and repeat against the packaged app.
9. **Refresh the submission.** Update PRD, README, privacy statement, judge instructions, screenshots, video script, evidence, package checksum, and Devpost story around the multi-agent town.

## Risks and safeguards

- **Bursting protocol events:** phase queues coalesce only visually; the safe activity timeline still records every meaningful event.
- **Five-session resource pressure:** one app-server owns all threads; start failures remain isolated per project and never cancel siblings.
- **Completion ordering:** the latest completed agent-message item is retained privately by thread, parsed when terminal completion arrives, then discarded before the safe event is emitted.
- **Interaction and ownership:** native input continues the active turn; terminal input resumes the same saved thread; post-turn Ghostty handoff is exclusive and explicitly reclaimed. Connection proof remains metadata-only.
- **Speech-bubble overlap:** each bubble is anchored inside a lot-specific lane with a fixed width and bounded text; the five-lot visual proof shows simultaneous bubbles without overlap.
- **Unsafe model summary:** marker content is untrusted and deny-by-default. Generic outcome/follow-up text is better than leaking a filename.
- **Demo duration:** the video starts multiple builders, follows one to completion, and uses the other active lots to prove concurrency without waiting for all five on camera.

## Approval boundary

The plan commits to five fixed lots, real independent sessions, persistent safe debriefs, and a stable one-canvas scheduler. Implementation touches the privileged session registry, persisted schema, public IPC contract, renderer state model, and entire Pixi scene. Those are the intended boundaries for this iteration.

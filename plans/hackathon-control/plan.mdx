# Codeville — Build Week Control Plan

Codeville is a desktop developer tool that turns real Codex sessions into a living village. A local repository is a house, a Codex thread is its resident builder, and actual planning, editing, command, approval, failure, and completion events drive the world. The hackathon deliverable is one complete, judge-testable vertical slice—not the future multiplayer platform.

<Callout id="deadline" tone="warning">

**Submission deadline:** Tuesday, July 21, 2026 at 5:00 PM PT. The official rules govern when the marketing page differs. Registration and the Devpost draft must be completed before the same deadline.

</Callout>

## Product outcome

A judge can launch the macOS application, choose an allowed repository, enter a concrete coding task, and watch a truthful visual representation of Codex doing the work. The resident changes activity only in response to real app-server events. When the turn completes successfully, the house receives a persistent upgrade. The application never sends repository contents, prompts, filenames, diffs, command output, or secrets to a multiplayer service.

The primary demo uses a safe fixture repository committed under `fixtures/demo-repo/`. The task is deterministic enough to demonstrate live work in under two minutes while still exercising real Codex planning, file changes, a command/test event, and completion.

## Scope

### Included

- One local project lot with a polished village scene and persistent house progression.
- Native folder selection, task entry, start, interrupt, and reset-demo controls.
- Local Codex app-server over stdio with version-matched generated TypeScript bindings where practical.
- A narrow `VillageEvent` translator that maps raw Codex protocol messages into safe game states.
- A secure activity panel for information that should not be represented only as animation.
- A deterministic demo mode that uses the real integration against the included fixture repository.
- Local JSON persistence under Electron's application data directory.
- macOS packaged build; documented development mode for other supported desktop platforms.
- Submission artifacts and evidence captured as first-class repository files.

### Explicitly deferred

- Public multiplayer, accounts, cloud synchronization, spectators, or remote machine control.
- Windows/Linux packaged release validation, auto-update, signing, notarization, or app-store distribution.
- A deep economy, free-roaming simulation, procedural buildings, generated runtime art, or multiple biomes.
- Uploading source code, prompts, commands, filenames, diffs, environment variables, logs, or terminal output.
- Replacing Codex approvals or weakening the user's configured sandbox.

<Callout id="scope-decision" tone="decision">

**Desktop decision:** use Electron + React + TypeScript + Vite + PixiJS. The machine has the exact Node/pnpm toolchain but no Rust toolchain, and app-server plus desktop process management is simplest when the shell, controller, tests, and UI share TypeScript. Tauri would add Rust and sidecar packaging risk without improving the judged vertical slice.

</Callout>

## User experience

1. The village opens on a calm project lot. The user chooses a repository or selects the included demo repository.
2. The house receives a project nickname. The task panel accepts one coding request and explains that execution remains governed by Codex sandbox and approval settings.
3. Starting the task creates a local thread. A resident walks to the house and changes stations as real events arrive: drafting table, workbench, inspection area, doorstep, or celebration spot.
4. The activity panel shows a coarse event timeline. Sensitive details are absent by default; any exact approval information remains local and explicit.
5. Success completes a visible building upgrade and records it locally. Failure leaves the property unchanged, shows a recoverable state, and preserves the safe event history for the demo.

### Process ownership

- `electron/main.ts` owns folder dialogs, app-server lifecycle, raw protocol input, persistence, and privileged IPC.
- `electron/preload.ts` exposes a small typed API; the renderer receives no unrestricted Node or filesystem access.
- `src/codex/translator.ts` is a pure, exhaustively tested raw-event-to-safe-event boundary.
- `src/state/session-machine.ts` is the deterministic session state machine used by both UI and tests.
- `src/game/Village.tsx` renders from safe application state; it never parses raw Codex messages.
- `src/features/task/TaskPanel.tsx` handles project/task controls and local approval surfaces.

## Ordered build

1. **Foundation:** pin Node 24.16.0 and pnpm 10.33.4, scaffold Electron/Vite/React/TypeScript, install the smallest dependencies, configure strict checks, and commit the planning/evidence baseline.
2. **Deterministic core:** implement the safe event contract, session state machine, local progression schema, and exhaustive unit tests before connecting Codex.
3. **Codex controller:** generate schemas from the installed CLI, implement stdio JSONL initialization/thread/turn handling, translate events, and test against protocol fixtures plus a real local smoke.
4. **Product shell:** build the task panel, village state, error recovery, secure IPC, persistence, and accessibility-friendly text equivalents.
5. **Game feel:** implement the cozy scene, resident stations, construction effects, status lighting, and completion upgrade. Animation remains a projection of state, never a timer pretending work occurred.
6. **Demo path:** create the fixture repo and a task that reliably exercises editing and tests within the video budget. Add reset controls and predictable starting data.
7. **Packaging and proof:** build the macOS artifact, run it against the fixture repository, capture live evidence, and repeat after every runtime-path fix.
8. **Submission:** complete README, test instructions, privacy statement, evidence log, Devpost copy, screenshots, video script, repository sharing requirements, and `/feedback` Session ID instructions.

## Failure handling

- Codex missing or too old: show the detected binary/version and exact remediation; never silently switch to canned events.
- Initialization or schema mismatch: stop the session, retain safe diagnostics locally, and show a retry action.
- Invalid/non-repository folder: explain the requirement before spawning Codex.
- User interruption: send `turn/interrupt`, transition to a neutral recoverable state, and grant no upgrade.
- Approval requested: pause visible construction until a local decision is returned; never auto-approve.
- Command/test failure: show failure state and allow Codex to continue; only the final successful completion grants progression.
- App crash/relaunch: restore committed progression, never an in-flight privileged process.
